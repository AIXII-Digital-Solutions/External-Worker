# ACYS Прогноз — модель, коэффициенты и вывод полей

Документ описывает, как считается прогноз: что за поле, откуда берётся и с каким коэффициентом. Данные
приходят из двух источников — справочник бортов/оценки (на каждый (борт, месяц): оператор, тип, рыночная
стоимость, кресла, лизинг, дата поставки) и история рейсов (реальные рейсы по бортам).

Два `Data Type`:
- **Actuals** — по одной строке на реальный рейс, история → **вчера**.
- **Forecast** — прогнозные рейсы, **сегодня → дата запроса + `FORECAST_HORIZON_YEARS`**.

---

## 0. Инварианты (жёсткие правила) и как они гарантируются

**Два якоря.** Старт прогноза = **`last_fact + 1`** (день после последнего факта) — так провала между фактами
и прогнозом нет НИКОГДА, даже если запрошенная дата ушла вперёд имеющихся данных. Контрактный год при этом
делится по **`as_of`** (день-в-день), а горизонт = **`as_of + FORECAST_HORIZON_YEARS`**. Если `as_of` позже
последнего факта, прогноз просто начинается сразу за фактами, а его контрактные годы режутся по дню `as_of`.

| # | Правило | Как обеспечено |
|---|---------|----------------|
| 1 | Факты — прошлое до последнего доступного дня | Actuals из истории рейсов; верхняя граница `first_seen < as_of`. FlightRadar грузится до **`today − 1`** (реальное вчера, текущий день не опрашивается; окно **не** привязано к as_of). Реестр покрытия продвигается только до реально имеющихся рейсов (свежие пустые дни остаются перезагружаемыми). Рейсы новее последней ревизии cirium атрибутируются **carry-forward** (последняя ревизия справочника), поэтому недавно летавшие борта не теряются. |
| 2 | Прогноз — сразу за фактами в будущее | Прогноз якорится на **`last_fact + 1`**; горизонт — `as_of + FORECAST_HORIZON_YEARS`. |
| 3 | Ни один день не может быть одновременно фактом и прогнозом | Факты `≤ last_fact`, прогноз `≥ last_fact + 1` → пересечений нет. |
| 4 | **Между фактом и прогнозом нет провала** | Прогноз стартует ровно на `last_fact + 1` — дыры нет при любой свежести данных (запрошенная дата не может утащить старт в будущее). Плюс floor `k≥1`, чтобы округление прората не выкинуло граничный месяц. |
| 5 | Стоимость в будущем **не растёт** | `slope = LEAST(0, regr_slope(...))` (зажат ≤0); проекция `aav` скоупится по оператору, иначе чужая поздняя строка того же рега давала бы отрицательный сдвиг → рост. |
| 6 | Списанный борт **не летит** | Белый список статусов + `_NOT_DEAD` анти-джойн: борт с любой строкой `Retired/Written off` под тем же идентификатором исключается, даже если есть устаревшая `In Service`. |
| 7 | Будущий флот **никогда не выбывает** | Флот = справочник (без списания по простою); floor `k≥1` для активного суб-флота **в каждом** месяце — тонкий флот (бизнес-джеты/вертолёты) не исчезает в сезонном провале; `latest`-ревизия берётся **по каждому plan_type** (иначе борта второго плана выпадали). |

Контрактный год везде считается по якорю `as_of` (перестамповка фактов + прогноз), поэтому обе половины отчёта
делят CY одинаково (якорный день — последний день своего CY: `16-сен → CY закрывается`, `17-сен → новый CY`).
`powerbi.z_dates_acys` берёт якорный день из **`max("Date")`** (горизонт = `as_of.day`), а НЕ из первой даты
прогноза — потому что прогноз теперь стартует на `last_fact + 1`, который может быть раньше `as_of`. Краевые
даты (29 фев високосного года) зажаты по длине месяца, чтобы `make_date` не падал.

## 1. Коэффициенты (все настройки)

| Константа | Значение | Смысл / зачем |
|-----------|----------|---------------|
| `HISTORY_START` | 2022-07-01 | Самый ранний рассматриваемый месяц (нижняя граница CY2022). |
| `FORECAST_HORIZON_YEARS` | 2 | Прогноз доходит до CY(`as_of.year + 2 − 1`); для запроса 2026 → **CY2027** (последний полный CY). |
| `LEVEL_L` | 3 | Текущий **уровень** = медиана последних 3 десезонализированных месяцев (≤ frontier). Короткое окно = следует за текущей активностью. |
| `SEAS_K` | 6.0 | **Сжатие** сезонного коэффициента к 1.0 по числу наблюдений: `seasonal[c] = (Σratio + K)/(n + K)`. Редкие месяцы тянутся к «нет сезонности». |
| `FRONTIER_FRAC` | 0.6 | Месяц «полный», если рейсов ≥ 0.6 × медианы окна (иначе он ещё догружается/отстаёт и в фит не идёт). |
| `FRONTIER_WINDOW` | 9 | Сколько последних месяцев берётся для медианы полноты. |
| `FORECAST_PAX_LOAD_FACTOR` | 0.8 | `Total PAX = Total Seats × 0.8` (предполагаемая загрузка). |

В прогнозе **нет ни потолка роста, ни правила списания по рейсам** — размер флота берётся прямо из
справочника (§3), поэтому рост и списания — ровно те, что в справочнике.

**Единицы измерения.** `Flight Time` и `Flight Time FR` — **десятичные часы**: `6.51` = 6 ч 31 мин, а *не*
интервал. Так они штатно суммируются и усредняются в BI (интервал не делает ни того, ни другого).
`Circle Distance` / `Actual Distance FR` — километры. Хранится полная точность (1 ч 42 мин 15 с → 1.704166…):
округление каждого рейса до суммирования увело бы налёт за контрактный год, поэтому округляем при
отображении, а не при хранении.

## 2. Контрактный год (день-в-день)

**12-месячное окно, заканчивающееся В якорный день**, помечается годом начала. Для запроса `10-июл-2026`:

```
CY2025 = (10-июл-2025 , 10-июл-2026]   т.е. 11-июл-2025 … 10-июл-2026
CY2026 = (10-июл-2026 , 10-июл-2027]
```

Правило для даты рейса `d` относительно якоря `(am, ad)`: `CY = год(d) − 1`, если `(месяц, день) ≤ (am, ad)`,
иначе `год(d)`. То есть якорный день — **последний** день своего CY (`10-июл-2026 → CY2025`; `11-июл → CY2026`).
И факт, и прогноз следуют этому по каждому рейсу, поэтому якорный месяц делится между двумя CY (Jul→Jul).

## 3. Модель прогноза — флот летает непрерывно

**Принцип:** каждый борт из **последней ревизии справочника** летает **каждый** прогнозный месяц; никто не
списывается, если справочник его не убрал; будущие поставки входят в тот месяц, на который приходится дата
поставки. Объём на борт — из типичной активности типа; маршруты — из типичной сети типа у **этой** АК
(никогда маршрут, которым она не летала).

### 3.1 Фит (на суб-флот = `Aircraft Sub Series`), по месяцам ≤ frontier

**Фолбэк истории.** Оператор может получить суб-серию, на которой **никогда не летал** (Air Arabia получает
12 `A321-253N neo ACF` при нуле рейсов на типе) — фитить нечего и маршрутного пула нет. Такой суб-флот
откатывается на историю своей **`Master Series`** — и фит, и пул маршрутов (`A321-253N neo ACF` → `A321neo`,
то есть реальная сеть оператора на `A321-251LR neo ACF`). Флот всегда остаётся целевой суб-серией;
заимствуется только *история*. Какая именно история взята, пишется в каждую строку
`acys_forecast_coefficients."History Key"` (§6) — подмена никогда не происходит молча. Суб-серия, у чьей
Master Series истории тоже нет, прогноза не получает.
- **`seasonal[1..12]`** — коэффициент месяца года. Для каждого календарного месяца `c`:
  `ratio_i = flights_i / mean(flights)`, затем `seasonal[c] = (Σ ratio + SEAS_K) / (n_c + SEAS_K)` (сжатие к 1.0).
- **`level`** — медиана последних `LEVEL_L` (=3) **десезонализированных** месячных сумм рейсов (`flights / seasonal[cal]`).
- **`base_fleet`** — медиана числа летавших бортов за последние `LEVEL_L` месяцев (тот флот, что дал `level`).

### 3.2 Frontier
`frontier` = самый свежий месяц с рейсами ≥ `FRONTIER_FRAC × median(последние FRONTIER_WINDOW месяцев)`. В фит
идут только месяцы ≤ frontier (отстающие свежие месяцы игнорируются для фита, но **остаются в actuals**).

### 3.3 На прогнозный месяц `m`, на суб-флот
- **`k` (рейсов на борт за месяц)** = `round(level / base_fleet × seasonal[cal(m)] × prorate)`.
  - `level / base_fleet` = десезонализированные рейсы **на один борт**.
  - `prorate` = покрытые-дни / дней-в-месяце = 1.0 для полных месяцев; **текущий** месяц покрывает сегодня → конец месяца (факт — более ранние дни), **последний** месяц покрывает начало → день `as_of + HORIZON`. Масштабирует и `k`, и распределение по дням одинаково.
- **`активный флот`** = борта суб-флота из последней ревизии справочника, входящие во флот к концу месяца `m`. **Никто не списывается** — борт уходит, только если его убрал справочник. Правила «что считать одним бортом» здесь важнее, чем кажется:
  - **Идентичность = (Serial Number, Aircraft Sub Series)**, а *не* Registration. У **заказанного планера регистрации ещё нет**, поэтому справочник ставит всем им заглушку — голый префикс страны (`A6-`). Ключ по Registration схлопывает **весь портфель заказов оператора — 113 бортов — в ОДИН** на суб-серию, и рост флота недосчитывается на порядок. Registration становится ключом только там, где серийника нет.
  - Справочник к тому же **не убирает строку `On order` после поставки**, поэтому один планер может присутствовать дважды (как заказ и как летающий, с одним серийником). Ключ «заказы по серийнику, флот по регистрации» посчитал бы его **дважды**; единый ключ по серийнику схлопывает его в один, а тай-брейк отдаёт победу **поставленной** строке (реальная дата поставки, а не оценка заказа).
  - **Белый список статусов — бортом считаются только эти восемь**, в двух корзинах. Корзина отвечает ровно на один вопрос: *если у строки нет даты поставки — борт уже летает или его ещё нет?*
    - **ЛЕТАЮЩИЕ** — `In Service`, `Storage`. Уже во флоте; отсутствие даты поставки означает лишь, что справочник её не записал, поэтому борт летает с первого месяца.
    - **ЗАКАЗАННЫЕ** — `On order`, `On option`, `LOI to Order`, `LOI to Option`, `Type swap`, `Reengineered`. Ещё не поставлены: борт входит во флот **в свой месяц поставки**, а **без даты поставки его нельзя разместить ни в одном месяце**, поэтому он отбрасывается, а не считается уже летающим.

    Всё остальное — **не флот**: `Cancelled` (не придёт никогда), `Written off` (разбит), `Retired` (списан), `Unknown`. Список **намеренно белый** — новый статус, появившийся в справочнике, не должен молча начать летать.

    То, что `Type swap` / `Reengineered` лежат в ЗАКАЗАННЫХ, а не в ЛЕТАЮЩИХ, — не формальность, а несущая конструкция: в последней коммерческой ревизии **у всех 3 490 строк `Type swap` даты поставки нет вообще**, а из 27 регистраций `Reengineered` в реальности не летает ни одна. В корзине ЛЕТАЮЩИХ они дали бы прогнозу 3 490 бортов с неизвестной датой прихода, каждый из которых летает *все* месяцы горизонта — ровно тот баг с фантомным флотом, который устроил `Cancelled`. В корзине ЗАКАЗАННЫХ они посчитаются в тот момент, когда справочник проставит им дату поставки (сегодня это 16 `Reengineered` и 0 `Type swap`).
  - **Регистрация в отчёте:** заглушка разворачивается в `Registration + Aircraft Sub Series + короткий серийник` → `A6-A320-251N neo-124349`. *Короткий* серийник — это всё после последнего дефиса: у заказа серийник в справочнике синтетический (`ABY-A320-124349`), и осмысленный в нём только хвостовой номер, а у поставленного борта лежит голый MSN (`13343`). Борт с настоящей регистрацией сохраняет её без изменений.
- **`пул маршрутов`** = рейсы **шаблонного месяца** суб-флота (последний тот же календарный месяц ≤ frontier; иначе последний месяц суб-флота). Это обычная сеть этого типа у оператора.
- **Генерация:** каждый активный борт делает `k` рейсов, циклически перебирая пул: `route = pool[((g−1) mod размер_пула) + 1]`, `g = 1..k`. Итого рейсов суб-флота = `k × |активный флот|` → растёт с поставками.

| Поле | Источник |

## 4. Agreed Value

База на (борт, месяц) = **Indicative Market Value (US$m)** из справочника. Правила:

1. **Wet-лизинг → 0** (`Lease Dry/Wet = 'Wet'`).
2. **Факт, пропущенный месяц** → перенос последнего известного значения **вперёд** (заполняет пробелы справочника); ведущие пропуски берут самое раннее известное.
3. **Прогноз** → проекция **собственной амортизации** борта вперёд: `value(m) = last_actual_value + slope × месяцев_после`, `slope = LEAST(0, regr_slope(value, месяц))` (**зажат ≤ 0 — не растёт**), пол 0.
4. **Новый борт** (никогда не летал → нет собственной истории оценок, а на непоставленный планер у справочника нет и рыночной стоимости) → **кросс-операторный бенчмарк типа**: среднее значение по такой же `Aircraft Sub Series` **по ВСЕМ операторам**, а не только по этому — горстка бортов одной АК это куда более тонкая выборка, чем все эксплуатанты типа. Берётся из **последнего месяца** справочника; если в этом месяце у типа нет ни одного оценённого планера — из **последнего года**. Держится плоско (амортизировать нечего — истории нет).

Три источника применяются по приоритету: **собственное спрогнозированное значение → собственное справочное значение → кросс-операторный бенчмарк типа**.

Порядок разрешения важен (и это защита от реального бага): SQL-функции `GREATEST`/`LEAST` **игнорируют** NULL-аргументы и возвращают NULL только если NULL все аргументы, поэтому `GREATEST(0, NULL)` даёт `0`, а не NULL. Без явной проверки на NULL новый борт молча превращается в «планер за $0», который выглядит как настоящее значение, и цепочка фолбэков не срабатывает вовсе.

### Средневзвешенное (`acys_summary_grouped`), на (борт, контрактный год)
| Колонка | Формула |
|---------|---------|
| Agreed Value on Inception | значение **первого** месяца CY |
| Agreed Value at End of Contract | значение **последнего** месяца CY |
| **Weighted Average Agreed Value** | **`(inception + end) / 2`** — среднее по времени прямой линии начало→конец, **всегда между ними** (устойчиво к внутригодовым скачкам рынка). |
| Activity-Weighted Average | `Σ(value × flights) / Σ(flights)` за CY (взвешено по рейсам). |

Wet-месяцы и неположительные значения исключаются из всех четырёх.

## 5. Вывод полей факта (для полноты)
Одна строка на реальный рейс. Атрибуты борта из справочника (достоверный оператор на (борт, месяц) = свежайшая
ревизия; wet-lease/ACMI-дубли схлопнуты). Маршрут/гео из источника рейсов. `Circle Distance` = большой круг
(haversine, км) между вылетом и прилётом; `Age = (Date − Delivery Date)/365.25`. Рейсы без кода аэропорта
вылета **или** прилёта отбрасываются. Нижняя граница по **дате**: `first_seen::date > (якорный день 2022)` —
рейс в якорный день при любом времени суток всё равно падает в CY2021 и отбрасывается, поэтому **CY2021 не появляется**.

## 6. `forecast.acys_forecast_coefficients` — коэффициенты для графиков

Одна строка на **(Operator, Aircraft Sub Series, Forecast Month)** — это и есть зерно прогноза. Пишется моделью
прогноза (обновление по оператору: удаляет свои строки оператора, затем вставляет заново). Содержит все
коэффициенты, стоящие за прогнозом. `Master Series` едет в строке только ради группировки на графиках.

| Колонка | Смысл |
|---------|-------|
| Operator | оператор |
| Master Series | широкий тип (для группировки на графиках) |
| Aircraft Sub Series | **ключ суб-флота**, по которому строится прогноз (фит / флот / маршруты) |
| **History Key** | **чья история легла в фит и маршрутный пул.** Равен `Aircraft Sub Series` — своя история суб-флота. Равен имени Master Series — **фолбэк** (§3.1): оператор никогда не летал на этой суб-серии, поэтому взята история его Master Series |
| Forecast Month / Calendar Month | 1-е число прогнозного месяца / месяц года (1..12, для сезонной кривой) |
| Frontier | последний полный факт-месяц (граница фита) |
| Level | десезонализированный текущий уровень рейсов (суб-флот) |
| Base Fleet | типичное число летавших бортов |
| Per Aircraft Rate | `Level / Base Fleet` = десезонализированные рейсы на борт |
| Seasonal Factor | `seasonal[Calendar Month]` |
| Proration | доля покрытых дней (1.0 кроме текущего и последнего месяцев) |
| Active Fleet | бортов летает в этом месяце (поставлены ≤ месяц) |
| Flights Per Aircraft | `k = round(Per Aircraft Rate × Seasonal Factor × Proration)` |
| Forecast Flights | `k × Active Fleet` — объём прогноза за месяц |
| Template Month | месяц-шаблон маршрутов |

Рецепты графиков:
- **Объём прогноза по времени:** `SELECT "Forecast Month", sum("Forecast Flights") … GROUP BY 1`.
- **Рост флота (приход поставок):** `sum("Active Fleet")` по `"Forecast Month"`.
- **Сезонная кривая:** `"Seasonal Factor"` по `"Calendar Month"` (distinct по `"Aircraft Sub Series"`).
- **Rate / level / base fleet:** одно значение на суб-флот.
- `Forecast Flights = Flights Per Aircraft × Active Fleet`, `Flights Per Aircraft = round(Per Aircraft Rate × Seasonal Factor × Proration)` — весь расчёт восстанавливается из строки.
