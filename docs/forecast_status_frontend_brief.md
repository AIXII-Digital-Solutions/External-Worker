# Бриф фронтенд-агенту: статус/прогресс прогноза (forecast job)

## Что изменилось (главное)

Раньше API отдавал прогресс «полосами» по шагам со случайными числами и почти без ETA, поэтому фронт **досочинял** динамику, чтобы бар не прыгал. Теперь **весь прогресс и оставшееся время полностью считаются и пушатся на стороне API** — на heartbeat’е ~каждые 2 секунды, поэтому `progress` и `eta` меняются вживую даже во время долгих шагов.

**Твоя задача меняется:** перестань выдумывать/синтезировать прогресс. Только **читай** значения от API и **визуально сглаживай рывки** между двумя присланными значениями. Ничего не придумывай сам.

## Откуда читать

- REST: `GET /status/{job_id}` — текущее состояние.
- SSE: `GET /status/stream` — live-поток тех же событий (Redis-канал `status:events`).

## Форма статуса

Верхний уровень:
- `state`: `queued` | `running` | `success` | `error` | `cancelled`
- `message`: короткий заголовок текущего шага (или финальное сообщение)
- `progress`: целое `0..100`. **Монотонно не убывает.** Значение `100` бывает **только** при `success`.

`payload` (JSON):
- `eta`: целое, **секунды до конца** (авторитетно). На терминальном статусе — `0`.
- `detail`: одно предложение-пояснение текущего шага (подзаголовок).
- `step`: `1..5` — номер текущего шага.
- `step_total`: `5`.
- `step_key`: `search` | `fetch` | `assemble` | `forecast` | `merge`.
- на `success` дополнительно едут сводные поля (`history_rows`, `forecast_rows`, `final_rows`, …).

## 5 шагов (в этом порядке)

1. **Searching historical data**
2. **Fetching data**
3. **Creating actuals dataset**
4. **Creating predictive analysis**
5. **Building dataset**
→ `success`

Заголовки/пояснения **не хардкодь** — рендери то, что прислал API (`message` + `payload.detail`). Номер шага можно показать как «Step 2 of 5» из `step`/`step_total`.

## Что делать (правила)

1. **Не синтезируй прогресс.** Убери любую собственную «придуманную» динамику/фейковое движение бара. `progress` и `eta` — источник истины и сами обновляются ~раз в 2 c.
2. **Сглаживай только визуально:**
   - Держи `displayedProgress`. На каждый пуш `p`: плавно анимируй `displayedProgress → max(displayedProgress, p)` за ~интервал heartbeat’а (≈2 c).
   - Держи `displayedEta`. На каждый пуш `eta`: цель = `eta`; между пушами уменьшай по реальному времени; на новом пуше **ре-синхронизируйся** к присланному значению.
3. **Никогда не показывай движение назад.** API гарантирует монотонность; если после реконнекта прилетело значение меньше показанного — зажми к максимуму виденного.
4. **Интерполяция не должна обгонять/противоречить API:** твой `displayedProgress` не должен превышать последний присланный `progress`, а `displayedEta` — не уходить в минус до нового пуша.
5. **Терминальные состояния:** на `success` — 100% и финальный `message`; на `error`/`cancelled` — останови анимацию и покажи финальный `message` (не продолжай крутить бар).

## Почему так надёжнее

- Прогресс — **взвешен по реальному времени** этапов, а не по номеру шага. Долгие этапы (fetch, прогноз по всем операторам) двигают бар и ETA по мере реальной работы.
- ETA — из скользящего среднего прошлых прогонов (самокалибрующегося) + замеров текущего прогона, а не из констант. Он может слегка «дёрнуться» на границе шага (оценка сменяется фактом) — это нормально, для того и нужно твоё сглаживание.
